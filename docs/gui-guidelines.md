# UNVET GUI 運用方針 (v1)

## 1. 目的

本ドキュメントは、UNVET の GUI を「目立たず堅実に運用するための補助UI」として定義し、
実装範囲と優先順位を明確化する。

## 2. 採用アーキテクチャ

- GUI 技術: Tauri (WebView ベース)
- コア処理: Rust 側で完結 (入力受信 / 正規化 / キャリブレーション / マッピング / 出力)
- 役割分離:
  - Rust: データプレーン (低遅延・高安定)
  - GUI: コントロールプレーン (状態表示・設定変更)

## 3. GUI スコープ (v1)

### 3.1 実装済み・運用必須

- Live Send トグル
- 出力クラッチホットキー設定
- 入力ソース切替 (iFacialMocap UDP / TCP, VMC / OSC UDP)
- VMC / OSC UDP 受信ポート変更
- 出力方式切替 (ETS2 / ATS, Relative Pointer, Absolute Pointer, Keyboard)
- 再キャリブレーション
- 接続状態と現在値表示 (yaw / pitch / confidence)
- 出力送信フィルター設定 (モード / プロセス一覧)

### 3.2 実装済み・調整系

- yaw/pitch 出力倍率
- yaw/pitch 反転
- Output Easing の ON/OFF と alpha

### 3.3 GUI 非対象 (設定ファイル管理)

- すべての詳細設定
- 実験パラメータ
- 運用上ほぼ変更しない内部しきい値
- 仮想 XInput などドライバー前提の機能設定

## 4. ログ・状態表示方針

- GUI は操作ログとランタイム警告を表示する
- エラー復旧操作は GUI から即時実行可能にする
- 接続状態・送信可否・active process を目視確認できるようにする

## 5. トレイ方針 (Windows)

- 初期優先度: 中
- 可能なら実装:
  - 最小化時にタスクトレイ格納
  - トレイから再表示 / 一時停止 / 終了
- 難易度が高い場合は v1 必須にしない

## 6. 性能・更新制約

- 制御ループと GUI 更新を分離する
- GUI 更新頻度は制限する (現状は約 120ms 間隔ポーリング)
- GUI 停止時もコア処理は継続できる構造を維持する

## 7. 実装順序 (推奨)

1. 最小 UI (入力/出力切替 + Live Send)
2. 調整 UI (倍率/反転/Easing)
3. 運用 UI (送信フィルター + ログ)
4. トレイ連携 (可能なら)
